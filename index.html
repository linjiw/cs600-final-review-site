If even the basics feel fuzzy

Three terms that get reused everywhere

Before flashcards, here are the three concepts that the rest of the course rests on. If any of these feel unclear, slow down here.

  1. Decision problem ≡ language. Every yes/no problem is a set of "yes" inputs. "Solving the problem" means: given $x$, output yes iff $x\in L$.
  2. Resource bound (time / space). A complexity class collects all languages that some Turing machine can decide using a bounded amount of a resource. Time = number of computation steps. Space = number of work-tape cells.
  3. Reduction. "$A$ reduces to $B$" via function $f$ means: $x\in A\iff f(x)\in B$, and $f$ itself is computable in some bounded resource (poly-time, logspace). If $A$ reduces to $B$ and $B$ is easy, then $A$ is easy.

Plain-English term dictionary

These are the words that make proofs sound harder than they are. Read the "exam move" column as what you should do when the term appears in a prompt.

Decision problem / language

A yes/no problem written as a set of yes-instances.

Exam move: translate "$x\in L$" to "input $x$ should be accepted."
Yes-instance / no-instance

A yes-instance belongs to the language; a no-instance does not.

Exam move: prove both directions, not just the accepting case.
Decider

An algorithm that always halts and answers accept/reject correctly.

Exam move: say why it halts within the claimed resource.
Oracle

A black-box subroutine you may query as if it answers instantly.

Exam move: count how many oracle calls you make.
Witness / certificate

The object proving a yes answer: assignment, path, coloring, vector.

Exam move: name its format before constructing or verifying it.
Reduction

A transformation from problem $A$ to problem $B$ preserving yes/no answers.

Exam move: prove $x\in A\iff f(x)\in B$ and bound resources for $f$.
Logspace reduction

A reduction whose transducer uses only $O(\log n)$ work space.

Exam move: output one local piece at a time; never store the full output.
Configuration

A full snapshot of a machine: state, head positions, and work tape.

Exam move: use configurations as graph vertices.
Configuration graph

Graph where edges are one-step machine moves.

Exam move: reduce machine acceptance to reachability from start to accept.
Deterministic vs nondeterministic

Deterministic has one next move; nondeterministic may guess among many.

Exam move: yes means some branch accepts; no means no branch accepts.
Complement / co-class

The same problem with yes and no swapped.

Exam move: be careful which answer is trustworthy.
Hard vs complete

Hard: everything in the class reduces to it. Complete: hard and also inside the class.

Exam move: prove membership and hardness separately.
Self-reducibility

Use a decision decider repeatedly to build the witness.

Exam move: force one witness piece and maintain extendability.
Fan-in

The number of inputs entering one gate.

Exam move: replace fan-in $k$ by a binary tree when moving to $\mathcal{B}_0$.
Size vs depth

Size counts gates; depth counts longest input-to-output chain.

Exam move: count both, because one can be small while the other is large.
Uniform vs nonuniform

Uniform families are generated by an algorithm; nonuniform families can be hardwired.

Exam move: for $\mathsf{P/poly}$, remember the family need not be computable.
One-sided error

Only one answer can be wrong; the other is always trustworthy.

Exam move: identify whether accept or reject is safe.
Amplification

Repeat independently to shrink error probability.

Exam move: multiply independent failure probabilities.
Cut / min-cut

A cut separates a graph into two sides; a min-cut uses the fewest crossing edges.

Exam move: fix one true min-cut and analyze the chance you never contract it.
Contraction

Merge two vertices connected by an edge and keep parallel edges.

Exam move: success in Karger means never contracting an edge of the fixed min-cut.

Section 03 Exam-critical

Self-reducibility

A decision algorithm only says yes or no. A search algorithm produces the witness that proves yes. Self-reducibility is the trick of squeezing a search algorithm out of a decision algorithm.

Why we care

The decision/search gap

You've spent the whole semester on decision problems — yes/no questions like "is this graph 3-colorable?" or "does this CNF have a satisfying assignment?". But in real life, knowing that a satisfying assignment exists is rarely useful: you usually want the assignment itself.

Self-reducibility says: for many problems, a yes/no decider is already enough to construct the witness. You don't need a separate "search algorithm." You just call your decider polynomially many times in a clever way.

Analogy
Imagine a sommelier who can only answer one question: "Does this case contain at least one bottle of red wine?" To find a red wine bottle, you ask once. If yes, split the case into two halves; ask which half has a red. Recurse on that half. After $\log_2(\text{case size})$ questions you've narrowed it down to one bottle. That's self-reducibility: a yes/no oracle plus a "split and force" trick gives you the actual object.
The core idea
Pin down one piece of the witness at a time. After each pin, check that the partial witness is still extendable — i.e. ask the decider on a restricted instance. Keep an accepting choice, and keep going.
The shape of every self-reducibility proof
Three things appear in every proof, in this order:
  1. Witness format. What pieces does the answer have? (a vector, a coloring, an assignment).
  2. Forcing gadget. A way to modify the input that says "the answer must use this value here."
  3. Invariant. A guarantee that, after each forcing step, some valid witness still exists.
If you remember those three slots, every self-reducibility problem becomes filling in three blanks.
Template F · Self-reducibility
  1. Let $D$ be a decider for the decision version.
  2. Run $D$ on the original instance. If it rejects, no witness exists.
  3. Build the witness piece by piece, maintaining: the part already fixed can still extend to a full witness.
  4. For each next piece, build restricted instances that force each candidate value, query $D$ on them, keep one accepting choice.
  5. Bound the number of queries by a polynomial.
Worked example · 3-COLOR self-reducibility

Step 1. Run the 3-COLOR decider $D$ on $G$. If reject, stop.

Step 2 — color anchors. Add three new vertices $r,g,b$ joined as a triangle. In any proper 3-coloring they get three different colors; call them Red, Green, Blue.

Step 3 — force one vertex. To force $v$ to take $r$'s color, add edges $v$–$g$ and $v$–$b$. Now $v$ can't use Green or Blue, so it must take Red.

Step 4 — sweep. For each original vertex $v_i$, try forcing Red, then Green, then Blue. Query $D$ each time. At least one choice must succeed (invariant). Keep it.

Step 5 — query budget. $1+3|V|$ queries — polynomial. Done.

Three graduated examples — same recipe, harder each time

Warm-up SAT (the simplest possible)

Decision: "Does CNF formula $\varphi$ have a satisfying assignment?" Witness: a Boolean assignment $x_1,\ldots,x_n\in\{0,1\}$.

  1. Run the decider $D(\varphi)$. If reject, no witness — stop.
  2. Try $x_1=0$: substitute $0$ for $x_1$ everywhere in $\varphi$, simplify, get $\varphi_0$. Query $D(\varphi_0)$.
  3. If accept, commit $x_1\leftarrow 0$, continue with $\varphi_0$.
  4. Else commit $x_1\leftarrow 1$, continue with $\varphi_1$.
  5. Repeat for $x_2,\ldots,x_n$.

Forcing gadget: substitution. Invariant: $\varphi_i$ is still satisfiable. Queries: exactly $n$. This is the cleanest example.

Standard — ILP ILP self-reducibility

$\mathsf{ILP}=\{\langle A,b,\ell,u\rangle:\exists\,x\in\mathbb{Z}^n,\ Ax\le b,\ \ell\le x_i\le u\}$. Witness: integer vector $x=(x_1,\ldots,x_n)$.

  1. Run $D$ once. If reject, no witness. Otherwise the invariant "some feasible solution exists" holds.
  2. Pin coordinates left to right. We will determine $x_1$, then $x_2$, etc. After fixing $x_1=a_1,\ldots,x_{i-1}=a_{i-1}$, the invariant is: some feasible solution still extends this prefix.
  3. Encode "$x_j=a_j$" using two inequalities. ILP allows only $\le$ constraints, so write $x_j\le a_j$ and $-x_j\le -a_j$. Add both rows to $A,b$ when fixing.
  4. Binary-search the next coordinate. For $x_i$, set $L=\ell, R=u$. While $L < R$: pick midpoint $m$, add the constraint $x_i\le m$, query $D$.
    • Accept → keep the lower half: $R\leftarrow m$.
    • Reject → must lie in upper half: $L\leftarrow m+1$, add $x_i\ge m+1$.
    When $L=R$, commit $x_i\leftarrow L$.
  5. Why binary search is polynomial. Numbers are written in binary, so $\log_2(u-\ell+1)$ is polynomial in input size. Each coordinate uses $O(\log(u-\ell))$ queries; total $O(n\log(u-\ell))$ queries.
Twist — 3-coloring 3-COLOR self-reducibility (the gadget twist)

Witness format: a function $\chi:V\to\{\text{Red},\text{Green},\text{Blue}\}$ such that no edge has equal endpoints. The twist: colors don't have a natural order, so we can't binary-search. We need a forcing gadget.

  1. Run $D$ once. If reject, no proper coloring exists.
  2. Build color anchors $r,g,b$. Add three new vertices and the triangle edges $rg,gb,br$. In any proper 3-coloring of the new graph, $r,g,b$ must each get a different color — call them Red/Green/Blue (which one is which doesn't matter, just that they're different).
  3. Forcing gadget. To force a vertex $v$ to take Red (= the color of $r$), add edges $vg$ and $vb$. Now $v$ is forbidden from getting Green or Blue, so it must be Red. (Symmetric for Green or Blue.)
  4. Sweep all vertices. For each $v_i\in V$: try forcing Red, then Green, then Blue. Query $D$ each time. At least one must accept (invariant: a proper extension exists). Keep one accepting choice, lock in those edges, move on.
  5. Query budget. $1$ initial $+ 3|V|$ trials = $O(|V|)$ queries — polynomial.

Compare with ILP: there the gadget was a numeric inequality; here the gadget is a forbidden-edge construction. Same recipe, different gadget.

Common mistakes
Wrong: "Try $\chi(v_1)=\text{Red}$. If $D$ rejects, the graph isn't 3-colorable." No — $D$ might reject just because that particular fix isn't extendable. You haven't ruled out 3-colorability; you've ruled out "$v_1=$Red" specifically.
Right: "Try Red, Green, Blue in turn. At least one must accept by the invariant; commit to one accepting choice and continue."
Wrong: "For ILP, just try every value $x_i\in[\ell,u]$ in order." That's $u-\ell+1$ queries — but $u-\ell$ can be exponential in input size since $\ell,u$ are written in binary. Always halve, never enumerate.
Right: Binary-search the interval. $O(\log(u-\ell))$ queries — polynomial.
Why does 3-COLOR self-reducibility add a triangle $r,g,b$?
Practice: To force vertex $v$ to take the color of $g$, which edges should you add?
Add edges $v$–$r$ and $v$–$b$. That blocks Red and Blue, leaving only Green available for $v$.

Logspace ($\mathsf{L}$) — tiny scratch paper

Picture a Turing machine with the input on a read-only tape and a tiny scratch tape, only big enough for a few counters. That's it. That's logspace.

Why we measure space at all

From "how long" to "how much memory"

So far we measured time (number of steps). Now we measure space (cells of the work tape). Why? Because the universe of "fast" looks different when you're memory-limited:

  • You can compute things in $O(\log n)$ space that you can not compute in $O(\log n)$ time (a logspace machine has plenty of time — up to $n^{O(1)}$ steps).
  • Space, unlike time, is reusable. You can clear a counter and use the same cells again.
  • $\mathsf{L}\subseteq\mathsf{P}$ (every logspace decider halts in polynomial time, by configuration counting). Whether the inclusion is strict — $\mathsf{L}\ne\mathsf{P}$ — is a famous open problem, alongside $\mathsf{P}\stackrel{?}{=}\mathsf{NP}$. Don't claim strictness on the exam.
Analogy
Think of logspace as "a sticky note next to a textbook." The textbook (input) is huge, you can flip back and forth as much as you want, but you can't write in it. Your sticky note (work tape) only fits a few small numbers. You're allowed to rescan and recompute as much as you want — but a halting logspace decider still has only polynomially many distinct configurations, so it terminates in polynomial time.
The single fact behind everything
A counter that holds a number from $0$ to $n$ uses $\lceil\log_2(n+1)\rceil$ bits in binary. So storing one counter is free in logspace, and a constant number of counters is also free. The line between "in $\mathsf{L}$" and "not in $\mathsf{L}$" is almost always: did I just store something whose size grows with $n$?
Template A · Prove $L\in\mathsf{L}$
  1. Say: read-only input tape, $O(\log n)$ work tape.
  2. Describe the algorithm in plain English.
  3. List every variable kept on the work tape.
  4. Bound each variable by $n$, so each uses $\log n$ bits.
  5. State the invariant.
  6. State the final accept condition.

Three graduated worked examples

Trivial "Even number of 1s" — constant space

$L=\{w\in\{0,1\}^*:\#_1(w)\equiv 0\pmod 2\}$.

Store one bit, $\mathit{parity}$. Initialize to $0$. Scan the input left to right; on each $1$, flip $\mathit{parity}$. Accept iff $\mathit{parity}=0$ at the end.

Total work-tape: 1 bit. That's $O(1)\subseteq O(\log n)$. Hence $L\in\mathsf{L}$. (Works for any regular language — every regular language is in $\mathsf{L}$, with $O(1)$ space, just by simulating the DFA.)

Standard "Equal 0s and 1s" — counters appear

$L=\{w\in\{0,1\}^*:\#_0(w)=\#_1(w)\}$.

Store two counters $c_0,c_1$, each initialized to $0$. Scan the input left to right; on $0$ increment $c_0$, on $1$ increment $c_1$. Accept iff $c_0=c_1$.

Each counter is at most $n$, so each fits in $\lceil\log_2(n+1)\rceil$ bits. Total: $O(\log n)$. Hence $L\in\mathsf{L}$. (Note: this language is not regular, so a DFA can't decide it — but a logspace TM can, with two counters.)

Twist — longest-run language "Longest a-run ≥ longest b-run"

$L'=\{w\in\{a,b\}^*:\text{longest a-run}\ge\text{longest b-run}\}$.

  1. State the model. "Read-only input tape. $O(\log n)$ work tape."
  2. Decide what variables to keep. We need: the best a-run seen so far ($\mathit{bestA}$), the best b-run ($\mathit{bestB}$), the current run length ($\mathit{curr}$), and which symbol we're currently running on ($\mathit{sym}$).
  3. Walk the input. For each character $x$:
    • If $x=\mathit{sym}$: increment $\mathit{curr}$.
    • Else: if $\mathit{sym}=a$, update $\mathit{bestA}\leftarrow\max(\mathit{bestA},\mathit{curr})$; if $\mathit{sym}=b$, update $\mathit{bestB}$. Then $\mathit{sym}\leftarrow x$, $\mathit{curr}\leftarrow 1$.
    At end of input: do one final update.
  4. Accept condition. Accept iff $\mathit{bestA}\ge\mathit{bestB}$.
  5. Space audit. $\mathit{bestA},\mathit{bestB},\mathit{curr}\le n$ → $O(\log n)$ bits each. $\mathit{sym}\in\{a,b\}$ → 1 bit. Total: $O(\log n)$. Done.

Common mistakes
Wrong: "Use a list of all the run lengths I've seen." That list has up to $n/2$ entries — that's $\Theta(n)$ space, way too much.
Right: Only keep the best so far, plus what you're tracking right now. Throw away history.
Wrong: "Copy the input to the work tape so I can re-scan it." The input is length $n$ — that violates the $O(\log n)$ bound.
Right: Keep the input on the read-only input tape; scan it again if you need to. The space bound only constrains the work tape, not how many times you re-read the input — a halting logspace decider still uses only polynomially many steps overall (configuration counting).
What does NOT fit
  • An array of $n$ booleans
  • The set of vertices reachable from $s$
  • An entire path you've traversed
  • A copy of the input
  • A list of run lengths
What DOES fit
  • A counter from $0$ to $n^k$ (any fixed $k$)
  • $O(1)$ pointers into the input
  • The vertex you are currently looking at
  • A "best so far" running max/min
  • A constant number of bits of state

Practice: "Strings over $\{0,1\}$ with the same number of 0s and 1s" — does the obvious approach work?
Yes — keep two counters $c_0,c_1$, each $\le n$. Each fits in $\log n$ bits. Accept iff $c_0=c_1$. (Note: this language isn't regular, but it is in $\mathsf{L}$.)
Section 05 Exam-critical

$\mathsf{NL}$ and $\mathsf{PATH}$

$\mathsf{NL}$ adds nondeterminism to logspace. The mental model: an $\mathsf{NL}$ machine guesses one step at a time and only remembers where it currently is.

Why nondeterminism + tiny memory is a class on its own

Same idea as $\mathsf{NP}$, but with space instead of time

Recall how $\mathsf{NP}$ works: there's a polynomial-time verifier, and we accept iff some polynomial-length witness causes it to accept. $\mathsf{NL}$ is the same idea, but the verifier must run in logspace:

  • $\mathsf{NP}$: witness has size $n^{O(1)}$, written down on a separate tape, verifier runs in time $n^{O(1)}$.
  • $\mathsf{NL}$: witness is generated one step at a time; verifier has only $O(\log n)$ space, so it can never store the whole witness.

The key restriction: an $\mathsf{NL}$ machine sees its random/guessed bits one at a time, then must decide what to do with $O(\log n)$ memory. It cannot "save up" guesses for later.

Analogy
Imagine playing a maze game where you can see your current room and you can guess which door to take next. You have a notepad with only enough space for the room you're in plus a step counter. You can't write down the full path. You just guess, move, repeat. If some sequence of guesses leads to the exit, you accept. That's exactly the $\mathsf{NL}$ machine deciding $\mathsf{PATH}$.

Yes-instances

At least one branch of the nondeterministic computation must accept.

No-instances

No branch may accept — every guess sequence must fail verification.

Template B · Prove $L\in\mathsf{NL}$
  1. Describe what the machine guesses nondeterministically.
  2. Yes-instance: some guess sequence verifies. ✓
  3. No-instance: no guess sequence verifies. ✓
  4. List variables. Bound each by $n$.
  5. Conclude $L\in\mathsf{NL}$.

$\mathsf{PATH}\in\mathsf{NL}$ — the canonical example

$\mathsf{PATH}=\{\langle G,s,t\rangle: \text{there is a directed path from }s\text{ to }t\}$.

Start at s. Store only the current vertex and a step counter (≤|V|).
Why can't an $\mathsf{NL}$ machine simply store the whole guessed path?
Section 06 Exam-critical

$\mathsf{PSPACE}$ & Savitch

$\mathsf{P}$ and $\mathsf{NP}$ measured time. Now we measure memory. The headline: polynomial space is robust under nondeterminism — Savitch only pays a square.

Why $\mathsf{PSPACE}$ is bigger than you think

Polynomial space, exponential time

A polynomial-space machine is allowed up to $n^k$ tape cells — that's a lot more than $\log n$. Why is it interesting? Because space is reusable: the same $n^k$ cells can be re-written as many times as you want, so the same machine can run for up to $2^{n^k}$ steps without repeating a configuration.

That makes $\mathsf{PSPACE}$ enormous: it contains all of $\mathsf{NP}$, and even problems where you have to think about strategies in two-player games (like generalized chess on an $n\times n$ board). $\mathsf{P}\subseteq\mathsf{PSPACE}$ is obvious — but $\mathsf{P}=\mathsf{PSPACE}$ is open and most likely false.

Analogy for Savitch
Imagine you're trying to verify "can I get from city $u$ to city $v$ in at most $T$ days?" using a single pad of paper. The naive approach (list every reachable city) needs lots of paper. Savitch's trick: ask "is there a midpoint city $m$ that I can reach from $u$ in $T/2$ days, and from which I can reach $v$ in $T/2$ days?" — recurse, re-using the same pad of paper. Each recursion only adds a constant amount of paper. Recursion depth is $\log T$, so total paper is $O(\log T)\times O(\text{paper per call})$.
Configuration counting
A configuration is a complete snapshot: state + head positions + work tape contents. A space-bounded decider can have at most $2^{O(f(n))}$ distinct configurations and must halt before repeating one. Hence $\mathsf{SPACE}(f)\subseteq\mathsf{TIME}(2^{O(f)})$.

The main inclusion chain

$\mathsf{L}\subseteq\mathsf{NL}\subseteq\mathsf{P}\subseteq\mathsf{NP}\subseteq\mathsf{PSPACE}=\mathsf{NPSPACE}\subseteq\mathsf{EXPTIME}$

Plain English: nondeterministic space simulates deterministically with only a square blow-up. Since polynomials are closed under squaring, $\mathsf{PSPACE}=\mathsf{NPSPACE}$.

Practice: explain in one paragraph why $\mathsf{NP}\subseteq\mathsf{PSPACE}$.
A poly-time NTM runs for at most $n^k$ steps on any branch, so it uses at most $n^k$ tape cells. Simulate it deterministically by trying every $n^k$-bit guess string in turn; reuse the same poly-size work tape. Total space: polynomial. Hence $\mathsf{NP}\subseteq\mathsf{PSPACE}$.
Section 07 Exam-critical

$\mathsf{PATH}$ is $\mathsf{NL}$-complete

Hardness is a logspace reduction from any $\mathsf{NL}$ language to $\mathsf{PATH}$ via the configuration graph.

What "complete" actually means here

Universal hardness via reductions

You saw the same pattern with $\mathsf{NP}$: we picked $\mathsf{SAT}$ and showed that every $\mathsf{NP}$ problem reduces to it. That made $\mathsf{SAT}$ $\mathsf{NP}$-complete: as hard as anything in $\mathsf{NP}$, give or take a poly-time reduction.

$\mathsf{NL}$-completeness is the same trick, scaled down: pick $\mathsf{PATH}$, show every $\mathsf{NL}$ problem reduces to it via a logspace reduction. We must use logspace — not poly-time — because if we used a more powerful reduction we could mash the whole problem into the reduction itself, making the notion of "complete for $\mathsf{NL}$" vacuous.

The configuration graph trick (reused everywhere)
Every space-bounded TM can be turned into a graph reachability problem. This trick appears in:
  • The $\mathsf{NL}$-completeness of $\mathsf{PATH}$ (this section).
  • The $\mathsf{NL}=\mathsf{coNL}$ proof (next section, with counting).
  • Savitch's Theorem (recursive midpoint search on the configuration graph).
  • Showing $\mathsf{NL}\subseteq\mathsf{P}$ (do BFS on the configuration graph).
Memorize the construction once and reuse it.
Analogy
Think of an $\mathsf{NL}$ machine $M$ on input $w$ as a board game: each "configuration" (state, head positions, work-tape contents) is a board position. Each transition is a legal move. Saying "$M$ accepts $w$" is the same as saying "starting from the initial board, you can reach a winning board through legal moves." But that's just graph reachability — i.e., $\mathsf{PATH}$.

Membership

$\mathsf{PATH}\in\mathsf{NL}$ — the walkthrough above.

Hardness

Take any $A\in\mathsf{NL}$ with NL-machine $M$. On input $w$, build $G_{M,w}$:

  • Vertices: configurations of $M$ on $w$.
  • Edges: one-step transitions.
  • $s$ = start config, $t$ = accept config.

$w\in A \iff \langle G_{M,w},s,t\rangle\in\mathsf{PATH}$.

Why the reduction is logspace
For an $\mathsf{NL}$ machine $M$ on input of length $n$, a configuration is (state, input head position, work-tape head position, work-tape contents). Each piece uses $O(\log n)$ bits, so a configuration fits in $O(\log n)$ space and the configuration graph has only polynomially many vertices — $n^{O(1)}$, not exponentially many. The logspace transducer iterates through pairs of configurations and writes each valid one-step edge to its output tape one at a time; it never stores the whole graph, only the two configurations it is currently comparing. The output is polynomial-size and the work tape stays $O(\log n)$.

$\mathsf{NL}=\mathsf{coNL}$ — the counting trick

Showing reachability is in $\mathsf{NL}$ is easy: guess a path. Showing non-reachability is the surprising part. Immerman–Szelepcsényi.

Why this theorem is shocking

Compare with the $\mathsf{NP}$ vs $\mathsf{coNP}$ situation

For $\mathsf{NP}$: showing a graph has a Hamilton cycle is easy (guess the cycle, verify). Showing a graph has no Hamilton cycle requires checking that every permutation fails — much harder. We believe (but cannot prove) $\mathsf{NP}\ne\mathsf{coNP}$.

For $\mathsf{NL}$: by the same intuition, we'd expect $\mathsf{NL}\ne\mathsf{coNL}$. But Immerman and Szelepcsényi (independently, 1987) proved they're equal. The trick: instead of storing the set of reachable nodes, store only its count.

Analogy
You want to prove "Alice is not in the building" using only a clipboard with room for one number. You can't store a list of everyone in the building. But suppose someone tells you exactly $c$ people are in the building. Then you can: nondeterministically guess each of the $c$ people one at a time, ask each one "are you Alice?", and if none of them is Alice, you're done. The clipboard never holds more than the running count of people checked so far.
The pivot
Don't store the set of reachable vertices — store only the count $c_i$ = number of vertices reachable from $s$ in $\le i$ edges. A count fits in $\log n$ bits. The set doesn't.

The high-level shape

  1. $\mathsf{PATH}$ is $\mathsf{NL}$-complete, so it's enough to show $\overline{\mathsf{PATH}}\in\mathsf{NL}$.
  2. Storing all reachable vertices is too big.
  3. Store $c_i$ instead — count of vertices reachable in $\le i$ steps.
  4. Compute $c_{i+1}$ from $c_i$ by sweeping all vertices and using nondeterministic certificates.
  5. Once you trust $c_{n-1}$, sweep again to verify $t$ isn't among the reachable vertices.
  6. Only counters and vertex names are stored — logspace.
What to write on the exam
  1. Define $\overline{\mathsf{PATH}}$.
  2. Note it suffices because $\mathsf{PATH}$ is $\mathsf{NL}$-complete.
  3. Storing all reachable vertices doesn't fit in logspace.
  4. Introduce $c_i$.
  5. Explain inductive step $c_i\Rightarrow c_{i+1}$.
  6. Conclude space is $O(\log n)$ counters and names.
Practice: why is it OK that the algorithm runs in polynomial (or worse) time?
$\mathsf{NL}$ is a space class, not a time class. Logspace already implies polynomial time anyway, but the definition only constrains the work tape. We're free to recompute reachability certificates on demand.

Boolean circuits

Different model: instead of one Turing machine for all input lengths, a separate circuit $C_n$ for each $n$. Inputs are bits. Gates are AND, OR, NOT. No cycles.

Why we need a second model at all

A computational model closer to hardware

Turing machines are abstract; circuits are essentially what you build out of physical AND/OR/NOT gates. Asking "how big a circuit do I need?" is asking "how much hardware would this take?" — a more concrete, more practical question.

More importantly, circuits give us new ways to separate resources:

  • Size = total number of gates = "how much hardware".
  • Depth = longest input-to-output path = "how many sequential stages".

Two circuits can have the same size but very different depths — and depth is what captures parallel time. That's a distinction Turing machines can't easily make.

Analogy
Picture an assembly line. Size = total number of workers. Depth = number of stations a single product passes through. With more workers in parallel (larger size), the line can be wider but no faster per product. To make the line faster, you have to shorten the depth. Same circuit, two independent dials.
Critical conceptual leap: one circuit per input length
A single circuit can only handle inputs of one fixed length $n$ — its inputs are the $n$ wires labelled $x_1,\ldots,x_n$. To decide a language $L\subseteq\{0,1\}^*$, we need an infinite family of circuits, one for each $n$: $$\mathcal{C}=(C_0,C_1,C_2,\ldots),\quad C_n(w)=1\iff w\in L\text{ for all }w\in\{0,1\}^n.$$ Allowing a different circuit for each length is what makes circuit families much more powerful than Turing machines — and what gives rise to the surprise that $\mathsf{P/poly}$ contains undecidable languages.

Size

Number of gates. Tracks "how big".

Depth

Longest path from input to output. Tracks parallel time if same-level gates fire together.

Majority circuit

Click the bits. Watch the OR-of-ANDs decide majority on the fly.

Why majority is the canonical example

A function that is "easy" for $\mathcal{B}_1$, harder for $\mathcal{B}_0$

Majority is the question: does at least half of my input equal 1? (We accept when ones are $\ge\lceil n/2\rceil$, i.e. ones $\ge k$ for $k=\lfloor n/2\rfloor+1$ when $n$ is odd, and $k=n/2$ when $n$ is even — in either case, $k=\lfloor n/2\rfloor+1$ witnesses a strict majority.) It has a nice OR of ANDs formula with $\binom{n}{k}\le 2^n$ terms, each an unbounded-fan-in AND — so over $\mathcal{B}_1$ (unbounded fan-in), it's exponentially big but only depth 2. Convert it to $\mathcal{B}_0$ (bounded fan-in) and the depth blows up to $O(\log\binom{n}{k})$ which is $O(n)$ in the worst case.

That contrast is the headline of this section: different gate bases give different size/depth trade-offs even on the same function.

Analogy for the "OR of ANDs" recipe
Imagine you want to know if a committee voted yes (majority of $n$ members say yes). For every possible "winning coalition" — every subset of size $k=\lfloor n/2\rfloor+1$ — ask: did all members of this coalition say yes? (one big AND). Then ask: did any of those coalitions all say yes? (one big OR). That's the circuit.
The key observation
Let $k=\lfloor n/2\rfloor+1$. Majority is 1 $\iff$ some set of $k$ positions are all 1. Hence $\mathrm{MAJ}_n(x)=\bigvee_{|S|=k}\bigwedge_{i\in S}x_i$ — an OR of $\binom{n}{k}$ ANDs.

Section 11 Exam-critical

$\mathsf{P/poly}$, uniformity, $\mathsf{NC}$, $\mathsf{AC}$

Three new ideas to keep straight

Non-uniformity, uniformity, and parallel depth

The whole circuit family discussion forks into three sub-ideas:

  1. Non-uniform circuits ($\mathsf{P/poly}$). Each $C_n$ can be designed however we like — even with help from a god. Surprisingly powerful: $\mathsf{P/poly}$ contains undecidable languages.
  2. Uniform circuits. A logspace transducer must be able to output $C_n$ on input $1^n$. This rules out cheating, brings circuits back in line with Turing machines, and is the right model for $\mathsf{NC}$ and $\mathsf{AC}$.
  3. Parallel depth ($\mathsf{NC}$, $\mathsf{AC}$). Once we have uniformity, measure depth (= parallel time). $\mathsf{NC}^i$ uses bounded fan-in; $\mathsf{AC}^i$ uses unbounded fan-in.
Why $\mathsf{P/poly}$ contains undecidable languages
Pick any unary undecidable language $L_K=\{1^n:n\in K\}$ for $K$ undecidable. For each length $n$, there's only one possible input ($1^n$). So $C_n$ is either "always accept" (if $n\in K$) or "always reject" — both trivial 1-gate circuits. The family has polynomial size. But the family itself (which $n$'s use which circuit) encodes the undecidable $K$. That's why we need uniformity — to forbid this kind of cheating.
The key inclusion you must be able to state
$$\mathsf{NC}^i\subseteq\mathsf{AC}^i\subseteq\mathsf{NC}^{i+1}.$$ First inclusion is by definition (bounded ⊆ unbounded). Second is fan-in conversion: an unbounded fan-in gate of size $k$ becomes a depth-$O(\log k)$ tree of bounded gates, and $k\le\text{poly}(n)$, so depth grows by at most an $O(\log n)$ factor.

$\mathsf{P/poly}$

Languages decided by polynomial-size circuit families. The family doesn't have to be computable — that's nonuniformity.

Surprising consequence: for any unary language, every length has only one possible input $1^n$. The circuit can be hardwired to accept or reject — even for undecidable unary languages.

Uniformity

A family is logspace-uniform if a logspace transducer can output $C_n$ given $1^n$. This rules out hiding uncomputable info in the family.

Binary addition $\in\mathsf{AC}^0$

Carries look sequential, but each carry can be written as a flat formula in the input bits. Define $g_i = a_i\wedge b_i$ ("generate") and $p_i = a_i\vee b_i$ ("propagate"). Then

$$c_{i+1}=\bigvee_{k=1}^{i}\Big(g_k\wedge\bigwedge_{j=k+1}^{i}p_j\Big).$$

Each carry is OR-of-AND with unbounded fan-in — depth 2 over $\mathcal{B}_1$. The sum bit $s_i$ is a constant-size formula in $a_i,b_i,c_i$. Hence addition is in $\mathsf{AC}^0$.

Section 12 Exam-critical

$\mathsf{RP}$, $\mathsf{coRP}$, $\mathsf{BPP}$, $\mathsf{ZPP}$

Randomized algorithms read random bits $r$ as well as the input $x$. The four classes differ only in what kind of error they allow.

Why bother with randomness

Sometimes a coin flip is faster than a deterministic plan

For some problems, a deterministic polynomial-time algorithm is hard or unknown to construct, but a randomized algorithm is easy. Famous example: primality testing (Miller–Rabin is fast, randomized; the deterministic AKS algorithm is slower and conceptually much more complex). Another example: randomized min-cut (Section 14).

The whole point of $\mathsf{RP}/\mathsf{coRP}/\mathsf{BPP}/\mathsf{ZPP}$ is to quantify what kind of error we tolerate. They are not different kinds of randomness; they are different error contracts.

Analogy — the four kinds of friend
Imagine four friends, each of whom you call to ask "is this restaurant good?":
  • $\mathsf{RP}$ friend: "If I say yes, trust me. If I say no, I might just be in a bad mood." (Never falsely accepts.)
  • $\mathsf{coRP}$ friend: "If I say no, trust me. If I say yes, I might be too generous." (Never falsely rejects.)
  • $\mathsf{BPP}$ friend: "I'm right at least 2/3 of the time, but either answer can be wrong sometimes."
  • $\mathsf{ZPP}$ friend: "If I give you an answer, it's correct. Sometimes I just say 'I don't know.'"
The relationships in one breath
$$\mathsf{P}\subseteq\mathsf{ZPP}=\mathsf{RP}\cap\mathsf{coRP}\subseteq\mathsf{RP}\subseteq\mathsf{BPP}\subseteq\mathsf{PSPACE}.$$ The two equalities you should know how to prove: $\mathsf{RP}\subseteq\mathsf{BPP}$ and $\mathsf{ZPP}=\mathsf{RP}\cap\mathsf{coRP}$.

Randomized inclusions — proof recipes

How to think about these proofs

Both proofs follow the same recipe (Template E)

Don't memorize a wall of probabilities. Memorize the recipe:

  1. Take the source machine (the machine guaranteed by the source class).
  2. Wrap it in a new machine that does something simple — usually run it 2–3 times and combine outputs.
  3. Compute the new yes-instance probability. Compare to the target class's threshold.
  4. Compute the new no-instance probability. Compare again.
  5. Conclude.

The "something simple" is usually one of: run twice and OR (for $\mathsf{RP}\to\mathsf{BPP}$), run twice and AND (for $\mathsf{coRP}\to\mathsf{BPP}$), or do "first machine, then second machine" (for the $\mathsf{ZPP}$ proof).

Theorem · $\mathsf{RP}\subseteq\mathsf{BPP}$

Let $L\in\mathsf{RP}$ with machine $M$. Define $B$: run $M$ independently twice, accept iff at least one run accepts.

No-instance. $M$ never accepts, so neither run accepts. $\Pr[B\text{ accepts}]=0\le \tfrac13$. ✓

Yes-instance. Each run rejects with probability $\le \tfrac12$. Both reject with probability $\le \tfrac14$. So $\Pr[B\text{ accepts}]\ge \tfrac34\ge \tfrac23$. ✓

Done. The simulator above is exactly this proof.

Theorem · $\mathsf{RP}\cap\mathsf{coRP}=\mathsf{ZPP}$

$(\subseteq)$. Let $M_R\in\mathsf{RP}$, $M_C\in\mathsf{coRP}$ both decide $L$. Define $Z$:

  1. Run $M_R(x)$. If it accepts → output 1. (Safe: $\mathsf{RP}$ never falsely accepts.)
  2. Run $M_C(x)$. If it rejects → output 0. (Safe: $\mathsf{coRP}$ never falsely rejects.)
  3. Otherwise → output $\bot$.

Yes-instance: $M_R$ accepts with prob $\ge \tfrac12$, so $\Pr[Z\text{ outputs }1]\ge \tfrac12$. Similar on no-instances. ✓

$(\supseteq)$. From a $\mathsf{ZPP}$ machine $Z$, build an $\mathsf{RP}$ machine (accept iff $Z$ outputs 1, otherwise reject) and a $\mathsf{coRP}$ machine (reject iff $Z$ outputs 0, otherwise accept).

The most common mistake on these proofs
Wrong: "By the law of large numbers, $\mathsf{BPP}$ machines are basically always correct." That's not a proof — it's a vibe. The exam wants you to give the actual probability bound, e.g. $(1/2)^2=1/4$ for two independent runs.
Right: Compute $\Pr[\text{both runs reject}]$ explicitly, multiply independent probabilities, and compare to the threshold $1/3$ or $2/3$.
Wrong: For $\mathsf{ZPP}$, "Run $M_R$ and $M_C$ in parallel and intersect their outputs." Vague — what counts as intersection?
Right: Be explicit: "Run $M_R$. If accept, output 1. Else run $M_C$. If reject, output 0. Else output $\bot$." Then state which step is safe and why (cite the one-sided guarantees of each source class).
Practice: an $\mathsf{RP}$ machine is run $m$ times. What's the false-reject probability on a yes-instance?
Each independent run rejects with probability $\le \tfrac12$. All $m$ reject with probability $\le 2^{-m}$. So false-reject $\le 2^{-m}$ — exponentially small in $m$.
Practice: prove $\mathsf{coRP}\subseteq\mathsf{BPP}$ from scratch (don't peek at the $\mathsf{RP}\subseteq\mathsf{BPP}$ proof first).
Mirror image. Let $L\in\mathsf{coRP}$ with machine $M$. Define $B$: run $M$ twice, accept iff both runs accept. Yes-instance: each run accepts with probability $1$, so both accept with probability $1\ge 2/3$. ✓ No-instance: each run accepts with probability $\le 1/2$, so both accept with probability $\le 1/4\le 1/3$. ✓ Hence $L\in\mathsf{BPP}$.
Practice: which step in the $\mathsf{ZPP}\supseteq\mathsf{RP}\cap\mathsf{coRP}$ direction uses the fact that $\mathsf{RP}$ never falsely accepts?
Step 1 ("if $M_R$ accepts, output 1"). The reason it's safe to commit to the answer 1 — without further checking — is precisely that an $\mathsf{RP}$ machine never falsely accepts a no-instance. Without that one-sided guarantee, this step would be wrong.

Randomized min-cut (Karger)

Pick a random edge. Contract it (merge endpoints, drop self-loops, keep parallel edges). Repeat until two vertices remain. Output the edges between them.

Why this algorithm is famous

Beautifully simple, surprisingly correct

A cut in a graph is a set of edges whose removal disconnects the graph into two pieces. The min-cut is the smallest such set. Classical deterministic algorithms (Stoer–Wagner, max-flow / min-cut) are clever and require care.

Karger's algorithm is the opposite: pick edges uniformly at random and contract them. It's almost embarrassing how simple it is. The miracle is the analysis — you can prove the random algorithm finds the min-cut with probability at least $2/(n(n-1))$. Run it a few hundred times and you're correct with overwhelming probability.

Why do we love it? It's the cleanest example in the course of "randomness as an algorithmic tool" — and the analysis introduces the technique of conditional probability telescoping that you'll see again in advanced courses.

Analogy
Imagine a network of pipes between cities, and you want to find the smallest set of pipes whose removal disconnects two specific cities. Instead of carefully tracing flow, Karger's strategy: pick a random pipe and "merge" its endpoints into one super-city, keeping all other pipes that touched either endpoint. Repeat until two super-cities remain. The pipes between them are your candidate cut. Because the min-cut is small (relative to total edges), the random pick rarely lands inside it, and the algorithm usually preserves it.
Common mistakes in the analysis
Wrong: "The probability of contracting a min-cut edge is $k/m$ where $m$ is the original number of edges." That bound is for the first iteration only; $m$ changes after each contraction.
Right: Use that every vertex has degree $\ge k$ at all times (the min-cut never changes), so a graph with $i$ vertices has $\ge ik/2$ edges. Hence $\Pr[\text{contract bad edge in iteration with $i$ vertices}]\le 2/i$.
Wrong: "Take $r=n$ runs and the failure probability becomes $(1-2/n^2)^n$, which is small." $(1-2/n^2)^n\to 1$ as $n\to\infty$. Not small.
Right: Take $r=cn^2\log n$ runs. Then failure probability $(1-2/n(n-1))^r\le e^{-2cr/n^2}=e^{-2c\log n}=n^{-2c}$. Pick $c$ for desired error.
Probability lower bound — the headline
Fix any true min-cut $C$ of size $k$. The algorithm succeeds iff it never contracts an edge of $C$. We will show $$\Pr[\text{success}]\ \ge\ \frac{2}{n(n-1)}.$$ The derivation below walks through it step by step.
Step-by-step derivation

The variable $i$ below is the number of vertices currently remaining at some stage of the algorithm. We start at $i=n$ and shrink to $i=2$.

  1. Min-cut survives across iterations. Any cut $C$ of the original graph corresponds to a cut of the contracted graph (parallel edges count separately). So the size of the global min-cut is still $\ge k$ at every stage, as long as we haven't contracted an edge of $C$.
  2. Every vertex has degree $\ge k$. If some vertex $v$ in the contracted graph had degree $
  3. Edges $\ge ik/2$. The handshake lemma: $\sum_v \deg(v)=2|E|$. With $i$ vertices each of degree $\ge k$, the sum is $\ge ik$, so $|E|\ge ik/2$.
  4. Bad edges are exactly $k$. "Bad" = an edge of the min-cut $C$. By assumption $|C|=k$ at the start, and (1) says $C$ remains a cut of size $k$ at every stage we have not yet ruined.
  5. Per-step bad probability $\le 2/i$. A uniform random edge picks bad with probability $$\Pr[\text{bad}]\ =\ \frac{|C|}{|E|}\ \le\ \frac{k}{ik/2}\ =\ \frac{2}{i}.$$ Equivalently, the chance we pick a safe edge is at least $1-\tfrac{2}{i}=\tfrac{i-2}{i}$.
  6. Multiply across all stages. The algorithm runs from $i=n$ down to $i=3$ (after that single step the graph has 2 super-vertices and we stop). Conditioning on having stayed safe so far, each stage contributes an $\ge \tfrac{i-2}{i}$ factor: $$\Pr[\text{success}]\ \ge\ \prod_{i=3}^{n}\ \frac{i-2}{i}.$$
Telescoping the product. Write out a few terms with $n=6$ to see the cancellation: $$\frac{1}{3}\cdot\frac{2}{4}\cdot\frac{3}{5}\cdot\frac{4}{6} \ =\ \frac{1\cdot 2\cdot 3\cdot 4}{3\cdot 4\cdot 5\cdot 6}.$$ The $3$ and $4$ in the numerator cancel the $3$ and $4$ in the denominator, leaving $\tfrac{1\cdot 2}{5\cdot 6}=\tfrac{2}{30}=\tfrac{2}{6\cdot 5}=\tfrac{2}{n(n-1)}$. In general, $$\prod_{i=3}^{n}\frac{i-2}{i} \ =\ \frac{1\cdot 2\cdot 3\cdots(n-2)}{3\cdot 4\cdot 5\cdots n} \ =\ \frac{(n-2)!/0!}{n!/2!} \ =\ \frac{2}{n(n-1)}.$$

So a single run succeeds with probability $\ge \tfrac{2}{n(n-1)}=\Theta(1/n^2)$ — small, but boostable: see the simulator below.

Part B · Short proofs

1. Prove "at least three 1s" $\in\mathsf{L}$.
One counter $c$ saturating at 3. Scan; if the symbol is $1$ and $c < 3$, increment. Accept iff $c=3$. Constant space ⊆ logspace.
2. Prove "path of length exactly 3 from $s$ to $t$" $\in\mathsf{L}$.
Loop over all $u,v$ using two vertex-name counters ($\log n$ each). Check the three edges $s\to u$, $u\to v$, $v\to t$ exist. Logspace.
3. Build a $\mathcal{B}_1$ circuit for "at least one input is 0".
$C_n(x)=\neg x_1\vee\cdots\vee\neg x_n$. $n$ NOTs feeding one unbounded OR. Size $O(n)$, depth 2.
4. Convert it to $\mathcal{B}_0$.
Keep the NOTs. Replace the fan-in-$n$ OR with a balanced binary OR tree: size $O(n)$, depth $O(\log n)$.
5. State the difference between $\mathsf{RP}$ and $\mathsf{coRP}$.
$\mathsf{RP}$ never falsely accepts (accept is trustworthy). $\mathsf{coRP}$ never falsely rejects (reject is trustworthy).
